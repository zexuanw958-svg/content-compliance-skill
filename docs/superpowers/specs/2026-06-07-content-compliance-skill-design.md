# 自媒体内容合规检测 Skill 设计

日期：2026-06-07

当前状态：2026-06-08 已完成第一版公开仓库封装。最新实现以仓库根目录 `README.md`、`content-compliance/SKILL.md`、`content-compliance/scoring.md` 和测试套件为准。

## 目标

构建一个可同时供 Codex / Code X、Claude Code、OpenClaw、Hermes Agent 使用的便携式 Skill，用于在自媒体内容生产过程中做平台定制化合规审查。

第一版聚焦两个平台：

- 抖音：普通发布、内容审核、DOU+ / 巨量引擎投放相关风险。
- 小红书：普通笔记、社区规范、薯条、蒲公英 / 聚光等商业化或投放相关风险。

Skill 的核心价值是把“感觉可能违规”转成可追溯、可解释、可修改的审查报告，帮助用户在选题前和成稿后降低发布与投放风险。

## 用户场景

### 选题闸门

用户在开工前输入平台和选题，Skill 判断该选题是否值得继续制作。

示例：

```text
平台：抖音
阶段：选题
选题：如何把国产模型接入 Codex
```

Skill 输出：

- 选题本身是否高风险。
- 可能触发的平台规则类型。
- 更稳妥的选题角度。
- 如果进入投放场景，风险是否会上升。

### 成稿审查

用户在写出口播稿、文案、画面描述后再次审查。

示例：

```text
平台：抖音
阶段：成稿
标题：国产模型接入 Codex，保姆级教程
口播稿：……
画面描述：
1. 打开 GitHub 项目页面
2. 展示下载两个开源项目
3. 终端执行安装命令
4. 字幕提示“去评论区拿链接”
```

Skill 输出：

- 总风险分和整体安全分。
- 10 格风险条 / 安全条。
- 分项安全仪表盘和最弱风险点。
- 风险清单。
- 命中的规则依据。
- 证据位置。
- 修改建议、保守改法和不建议使用的表达。

## 第一版范围

第一版支持：

- 平台：抖音、小红书。
- 阶段：选题闸门、成稿审查。
- 输入材料：选题、标题、口播稿 / 文稿、画面描述、是否计划投放。
- 输出：结构化报告、1-10 总风险分、1-10 整体安全分、10 格风险条 / 安全条、分项安全仪表盘、最弱风险点、规则依据、修改建议、免责声明。
- 规则来源：官方规则优先，规则卡必须能追溯到官方来源。
- 安装说明：根目录 `README.md` 覆盖 Codex / Code X、Claude Code、OpenClaw、Hermes Agent。
- 公开展示：根目录 README 使用 `assets/readme-banner.png`，原始大图在 `assets/readme-banner-source.png`，再生成提示词在 `assets/readme-banner.prompt.md`。

第一版不支持：

- 自动识别图片、截图或视频帧。
- 自动保证发布、审核或投放通过。
- 替代法律、广告、医疗、金融等专业审查。
- 自动实时同步所有平台规则。

## 官方规则调研要求

规则库必须采用 official-first 原则。

“完全拉取”的定义：

- 每个目标平台的官方规则入口都要进入 `references/sources.md`。
- 对能公开访问的官方页面，记录 URL、标题、访问日期、页面更新时间、适用范围和抓取状态。
- 对动态渲染或需要登录的官方页面，使用真实浏览器访问并记录可见内容范围。
- 对允许保存的页面，保留本地调研快照供内部整理使用。
- 对无法访问、需要登录、反爬阻断或不允许保存的页面，记录失败原因和下一步人工复核方式。
- 公开发布的 Skill 不转载官方规则全文，而发布结构化规则卡、来源链接和访问日期。

要求：

- 优先使用平台官方站点、官方帮助中心、官方规则中心、官方 App 内规则中心、官方商业化平台文档。
- 第三方文章只能用于发现入口，不能作为正式规则依据。
- 每一张规则卡必须包含至少一个官方来源链接或标记为 `needs_review`。
- 如果官方页面是动态渲染页面，必须使用真实浏览器访问并记录页面标题、访问日期和可见规则范围。
- 如果官方规则不可公开抓取或需要登录，应记录访问状态，不得凭空补全。
- 平台规则会变化，所有来源必须记录访问日期和更新时间。


## 审查流程

### 步骤 1：收集上下文

Skill 尝试识别：

- 平台：抖音或小红书。
- 阶段：选题或成稿。
- 材料：选题、标题、文稿、画面描述。
- 是否计划投放：DOU+、薯条、蒲公英、聚光或其他商业化推广。

如果平台缺失，Skill 只问一次平台。

如果阶段缺失：

- 只有选题时，默认运行选题闸门。
- 有标题、文稿或画面描述时，默认运行成稿审查。

### 步骤 2：加载规则

根据平台加载对应规则包：

- 抖音：`rules/douyin.md`。
- 小红书：`rules/xiaohongshu.md`。

如果用户计划投放，启用投放相关规则和发布场景系数。

### 步骤 3：分层审查

审查维度：

- 选题风险：题材是否天然高危，是否涉及强监管领域或敏感公共议题。
- 标题 / 封面风险：是否夸大、诱导、绝对化、制造对立或暗示违规收益。
- 文稿 / 口播风险：是否有违规承诺、导流、虚假宣传、误导性教程。
- 画面 / 字幕风险：是否展示二维码、下载链接、私信、评论区领取、第三方跳转、敏感素材。
- 商业化 / 投放风险：是否涉及资质、广告法、平台投放审核、内容合作披露。

### 步骤 4：评分

按评分算法计算每个风险项，再给出总风险分、整体安全分、风险条、安全条和分项安全仪表盘。

### 步骤 5：输出报告

报告包含：

- 基本信息。
- 总体结论。
- 总风险分、整体安全分、风险条、安全条。
- 分项安全仪表盘。
- 最弱风险点。
- 风险清单。
- 修改建议。
- 投放提醒。
- 免责声明。

## 风险评分算法

每个风险项先计算单项分：

```text
item_score =
规则严重度 × 命中置信度 × 曝光范围 × 发布场景系数 × 修正难度
```

### 规则严重度 severity

```text
1 = 轻微表达 / 格式问题
2 = 可能影响审核或推荐
3 = 明确平台不鼓励或限制
4 = 明确高风险规则，如导流、虚假承诺、资质缺失、商业推广违规
5 = 极高风险，如违法违规、诈骗、危险行为、严重伤害、重大敏感内容
```

### 命中置信度 confidence

```text
0.4 = 弱疑似
0.7 = 明显可疑
1.0 = 明确命中
```

### 曝光范围 exposure

```text
1.0 = 局部一句话 / 一个镜头
1.2 = 标题、封面、开头、结尾 CTA
1.5 = 多处重复出现，或是内容核心卖点
```

### 发布场景 scenario

```text
1.0 = 普通发布
1.2 = 有商业化、引导互动、评论区 / 私信 / 资料领取等动作
1.4 = 投放场景，如 DOU+、薯条、广告审核
```

### 修正难度 fix_difficulty

```text
1.0 = 改一句话即可
1.2 = 需要调整脚本结构或画面设计
1.5 = 选题本身高风险
```

### 总分

```text
max_item_score = 所有风险项中的最高 item_score
base = ceil(max_item_score / 15.75 * 10)
base = clamp(base, 1, 10)
accumulation = 非最高风险项带来的累积加成，封顶 2 分
final_score = min(10, round(base + accumulation))
```

说明：15.75 是单项分理论上限，来自 `5 × 1.0 × 1.5 × 1.4 × 1.5`。

累积加成规则：

```text
严重度 1-2 的额外风险：每项 +0.25
严重度 3 的额外风险：每项 +0.5
严重度 4 的额外风险：每项 +0.75
严重度 5 的额外风险：每项 +1.0
accumulation 最高为 2
```

分数解释：

```text
1-2：低风险，可发布
3-4：轻中风险，建议微调
5-6：中风险，建议修改后发布
7-8：高风险，不建议直接发布
9-10：极高风险，建议换选题或重写
```

报告必须说明评分依据，不允许只给分数。

### 可视化评分

报告必须同时包含风险视角和安全视角：

```text
总风险分：1-10，越高越危险。
整体安全分：11 - 总风险分，越高越安全。
风险条：固定 10 格红黄绿危险刻度，始终显示 `🟩🟩🟩🟩🟨🟨🟥🟥🟥🟥`，并标出当前风险分落点；不要把风险条做成绿色填充进度条。
安全条：固定 10 格安全余量条，填充格数等于整体安全分。
```

分项安全仪表盘固定包含：

```text
topic
title_or_cover
script_or_oral_wording
visual_or_subtitle
external_guidance_download_comment_private_message_or_qr
promotion_or_ad_review
regulated_industry
```

报告必须列出分数最低的 1-3 个最弱风险点，附证据位置、弱点原因、快速改法和保守改法。这样可以避免“整体分还行”掩盖局部高风险话术。

## 免责声明

报告必须包含以下免责声明，不能省略：

```text

```

## 设计约束

- 不凭空生成平台规则。
- 不把第三方经验文当正式依据。
- 不承诺过审。
- 不输出“必过”“一定安全”等绝对判断。
- 对缺少官方来源的规则必须标记 `needs_review`。
- 对平台审核不确定性必须显式说明。
- 对高风险领域必须建议专业复核。

## 验收标准

- 根目录 README 为中文用户优先，并说明 Codex / Code X、Claude Code、OpenClaw、Hermes Agent 的安装和使用方式。
- 用户能用自然语言说明“我要发抖音 / 小红书”，Skill 能选择正确规则包。
- 支持 `/检测` 和 `/content-compliance` 两个斜杠入口。
- 用户只给选题时，Skill 运行选题闸门。
- 用户给文稿或画面描述时，Skill 运行成稿审查。
- 输出报告包含 1-10 总风险分、整体安全分、10 格风险条 / 安全条和评分依据。
- 输出报告包含分项安全仪表盘和最弱风险点。
- 每个正式风险项必须关联规则 ID 和官方来源。
- 无法关联官方来源的风险只能标记为待复核提示，不能输出为确定命中。
- 报告必须包含免责声明。
